# ADR-001: Płaska struktura plików (bez katalogu `public/`)

## Status
Zaakceptowane

## Context

Specyfikacja wymaga dostarczenia dokładnie 3 plików: `index.html`, `app.js`, `styles.css`. Aplikacja nie wymaga build toolingu, testów, skryptów ani `package.json`. Istniejące projekty demo w repozytorium stosują dwa wzorce:

1. **demo-pomidoro**: Wszystkie pliki runtime w `public/` — `index.html`, JS, CSS. Na poziomie roota: `package.json`, testy, skrypty, dokumentacja agentów.
2. **demo-traffic-simulator**: `index.html` na poziomie roota, JS i CSS w `public/`. Brak `package.json`, brak testów.

Nowy projekt `demo-greeting` jest prostszy niż oba — ma tylko 3 pliki i zero toolingu.

## Decision

Zastosować strukturę płaską: wszystkie 3 pliki (`index.html`, `app.js`, `styles.css`) umieszczone bezpośrednio w `demo-greeting/`, bez podkatalogu `public/`.

```
demo-greeting/
  index.html
  app.js
  styles.css
```

## Alternatives Considered

### ALT-1: Root HTML + public/ (wzorzec traffic-simulator)
```
demo-greeting/
  index.html
  public/
    app.js
    styles.css
```
- **Za**: Spójność z `demo-traffic-simulator`, separacja HTML od zasobów
- **Przeciw**: Dodaje niepotrzebny katalog dla projektu z 3 plikami; ścieżki w HTML muszą wskazywać `./public/app.js` zamiast `./app.js`
- **Ocena**: Akceptowalna alternatywa, ale nadmiarowa

### ALT-2: Wszystko w public/ (wzorzec pomidoro)
```
demo-greeting/
  public/
    index.html
    app.js
    styles.css
```
- **Za**: Spójność z `demo-pomidoro`, przygotowane na dodanie `package.json` i testów
- **Przeciw**: Użytkownik musi nawigować do `public/`, żeby otworzyć `index.html`; spec mówi „flat directory"; nie ma czego separować na roota
- **Ocena**: Odrzucone — niepotrzebna złożoność

## Consequences

### Pozytywne
- Najprostsza możliwa struktura — spec spełniony literalnie
- Zero konfiguracji ścieżek — `./app.js` i `./styles.css` w HTML
- Otwarcie `index.html` bezpośrednio z eksploratora plików działa natychmiast
- Łatwa do zrozumienia dla nowych kontrybutorów

### Negatywne
- Jeśli w przyszłości QA zażąda testów, trzeba będzie dodać `tests/` i `package.json` — struktura urośnie. Nie jest to problem — migracja jest trywialna (dodanie plików nie zmienia istniejących).
- Brak spójności 1:1 z istniejącymi demo (oba używają `public/`), ale różnica jest uzasadniona skalą projektu.

## Notes
- Decyzja oparta na analizie wzorców z researchu (`research-existing-patterns.md`), który rekomenduje OPT-1 (flat structure).
- Spec jawnie stwierdza: „3 files: index.html, app.js, styles.css in demo-greeting/ directory".
- Jeśli zakres projektu urośnie (testy, build), ADR można zaktualizować i przenieść pliki do `public/`.
